# v2.1.0 真实场景测试用例

## 测试目标

验证 RuntimeContract foundation 在真实本地后端、真实 runtime persistence 和 Web 控制台下可重复读写，并确认 System Validation 页面具备稳定自动化锚点与 checkpoint 安全 readout。

## 前置条件

- 已启动启用 runtime persistence 的 Athena 后端。
- 已执行 `go run . migrate`。
- Web 控制台已启动，并能访问同一个后端。
- 若执行 DOM 路径，本机 Python 环境已安装 Playwright 与 Chromium。

## API 验证步骤

1. 确认后端健康：

```bash
curl http://127.0.0.1:8090/healthz
```

预期结果：返回健康状态。

2. 执行 RuntimeContract foundation smoke：

```bash
python3 scripts/control_plane_runtime_foundation_smoke.py --base-url http://127.0.0.1:8090
```

预期结果：

- `status` 为 `ok`
- `foundation.contracts` 大于 0
- `foundation.task_types` 大于 0
- `foundation.hook_bindings` 大于 0
- `records.steps`、`records.lifecycle`、`records.traces`、`records.usage`、`records.projections` 均大于 0
- `records.checkpoints` 存在且为数字；当有 checkpoint 时，返回项不得包含 raw `payload` 或 `resume_token`

## Web 验证步骤

1. 启动 Web 控制台并指向同一个后端。
2. 执行 DOM smoke：

```bash
python3 scripts/control_plane_runtime_foundation_smoke.py --base-url http://127.0.0.1:8090 --web-url http://127.0.0.1:5173
```

预期结果：

- `status` 为 `ok`
- `web.dom` 为 `ok`
- System Validation tab、Runtime Persistence Readout、checkpoint 安全摘要、Runtime foundation snapshot、capability surface、编辑器和验证按钮都能被稳定定位。

## 失败排查

- 若 `healthz` 失败，先检查后端端口、配置环境变量和数据库连接。
- 若 smoke 返回鉴权失败，检查是否配置了 control-plane token，并通过 `--control-plane-token` 传入。
- 若 foundation 计数为 0，检查 runtime bootstrap 是否执行，以及数据库 schema 是否指向正确 search path。
- 若 checkpoint readout 为空，先确认目标 run 是否产生等待态或 checkpoint ref；空 `items` 是允许的 fallback，但字段中不能出现 raw payload 或 resume token。
- 若 DOM smoke 失败，先确认 Web 控制台能打开 System Validation tab，再检查 `data-testid` 是否被移除或重命名。
- 若 Playwright 启动失败，重新安装 Python Playwright 浏览器依赖。
