# 本地开发与服务启动

## 这份文档解决什么问题

这份文档面向新进入仓库的开发同学，也面向需要帮你起服务的 AI。

目标只有两个：

- 让 Athena 在本机稳定启动
- 让开发者知道下一步去哪里继续调试、接模型和开控制面

## 推荐启动路径

本地启动分两档：

1. 最快启动：
   - 只验证服务是否能跑起来
   - 使用内存 session store + 内存 model store
   - 不依赖 PostgreSQL
2. 完整开发：
   - 验证 session 持久化、模型治理、control-plane 和 Web 控制台
   - 使用 PostgreSQL

新同学第一次进入仓库，建议先跑第 1 档。

## 环境要求

- Go `1.26`
- Node.js `22`
- npm
- Docker
- 可选：
  - PostgreSQL `14+`

## 启动前必须知道的事实

- 进程入口只有 4 个命令：
  - `go run . api-server`
  - `go run . migrate`
  - `go run . healthcheck`
  - `go run . version`
- 配置加载顺序固定为：
  1. `config/config.yml`
  2. `config/config.<APP_ENV>.yml`
  3. 环境变量覆盖
- `api-server` 启动时不会自动迁移数据库。
- 仓库里的 `config/config.yml` 只可视为当前开发默认值，不应直接假设适合你的本机环境。

## 方案 A：最快启动

这个方案用于验证服务活性、HTTP 路由、Swagger、control-plane 基本页和 AI 协作脚本入口。

### 1. 设置最小环境变量

```bash
export SESSION_STORE_DRIVER=memory
export MODEL_STORE_DRIVER=memory
export SECURITY_ENCRYPTION_KEY=athena-local-dev-key
export HTTP_PORT=8090
export CONTROL_PLANE_ALLOWED_ORIGINS=http://127.0.0.1:5173
```

说明：

- `SESSION_STORE_DRIVER=memory`
  - 避免本地第一次启动就卡在 PostgreSQL。
- `MODEL_STORE_DRIVER=memory`
  - 允许先起服务，不要求先准备 provider/model 表。
- `SECURITY_ENCRYPTION_KEY`
  - 即使是内存模式，也建议显式给一个本地值，避免后续切换存储模式时忘记补。

### 2. 启动服务

```bash
go run . api-server
```

### 3. 验证服务

```bash
curl http://127.0.0.1:8090/healthz
curl http://127.0.0.1:8090/swagger/openapi.json
curl http://127.0.0.1:8090/api/control-plane/bootstrap
```

预期：

- `/healthz` 返回 `{"status":"ok"}`
- `/swagger/openapi.json` 能返回 OpenAPI
- `/api/control-plane/bootstrap` 能返回控制面 bootstrap payload

## 方案 B：完整开发模式

这个方案用于验证：

- PostgreSQL session store
- PostgreSQL model store
- 显式 `migrate`
- 控制面配置持久化
- Web 控制台联调

### 1. 准备 PostgreSQL

如果本机没有 PostgreSQL，先用 Docker 起一个最小实例：

```bash
docker run --name athena-pg \
  -e POSTGRES_DB=secure_digital \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 \
  -d postgres:16
```

创建 schema：

```bash
docker exec -it athena-pg psql -U postgres -d secure_digital -c 'CREATE SCHEMA IF NOT EXISTS athena;'
```

### 2. 设置环境变量

```bash
export SESSION_STORE_DRIVER=postgres
export MODEL_STORE_DRIVER=postgres
export SECURITY_ENCRYPTION_KEY=athena-local-dev-key
export DB_TYPE=postgres
export DB_HOST=127.0.0.1
export DB_PORT=5432
export DB_NAME=secure_digital
export DB_USERNAME=postgres
export DB_PASSWORD=postgres
export DB_CONFIG='sslmode=disable search_path=athena client_encoding=UTF8'
export CONTROL_PLANE_ALLOWED_ORIGINS=http://127.0.0.1:5173
export SHARED_ROOT_DIR=$PWD/shared
```

### 3. 先迁移，再启动

```bash
go run . migrate
go run . api-server
```

## Web 控制台启动

后端起好后，再开前端：

```bash
cd web
npm install
npm run dev
```

默认地址：

- Web 控制台：`http://127.0.0.1:5173`
- 后端 API：`http://127.0.0.1:8090`

Vite 默认会代理：

- `/api` -> `http://127.0.0.1:8090`
- `/swagger` -> `http://127.0.0.1:8090`

如需覆盖默认代理目标，可在启动前设置：

```bash
export ATHENA_WEB_PROXY_TARGET=http://127.0.0.1:8090
```

## 本地对话为什么可能仍然不可用

服务能启动，不代表模型已经可用。

Athena 当前的模型治理是运行时配置，不是靠固定环境变量直接完成的。要让 `chat/respond` 和 `chat/stream` 真正出模型结果，还需要在服务启动后通过治理接口或控制面创建 provider/model。

最小路径：

1. 打开控制面
2. 进入 models 页面
3. 新增 provider
4. 新增 provider model，并设置 default/fallback
5. 再调用 `/api/chat/respond` 或 `/api/chat/stream`

## 给 AI 的最小指令模板

如果你希望 AI 按本文直接帮你起服务，可以直接给它下面这类指令：

```text
请按 docs/本地开发与服务启动.md 的方案 A 启动 Athena，
验证 /healthz 和 /api/control-plane/bootstrap，
如果缺环境变量就只用内存模式，不要接 PostgreSQL。
```

如果你已经有本机 PostgreSQL：

```text
请按 docs/本地开发与服务启动.md 的方案 B 启动 Athena，
先执行 migrate，再启动 api-server，
最后帮我验证 healthz、swagger 和 web 控制台联通性。
```
