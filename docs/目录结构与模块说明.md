# 目录结构与模块说明

## 这份文档解决什么问题

这份文档是 Athena 的目录级导航，目标是让新人和 AI 快速回答：

- 代码应该从哪里读起
- 某个能力应该落在哪个模块
- 哪些目录是主线，哪些目录是专项增强

## 顶层目录

```text
athena/
├── main.go
├── config/
├── internal/
├── web/
├── docs/
├── scripts/
├── deploy/
├── plugins/
├── Dockerfile
├── README.md
└── AGENTS.md
```

### 顶层职责

- `main.go`
  - 进程级命令分发入口，负责 `api-server / migrate / version`。
- `config/`
  - 基础配置文件，配合环境变量组成最终运行时配置。
- `internal/`
  - Athena 主业务代码。
- `web/`
  - 独立 control-plane Web 控制台。
- `docs/`
  - 当前真相、架构、API、启动和部署文档。
- `scripts/`
  - 验证、联调、打包和部署辅助脚本。
- `deploy/`
  - 云端部署样例、systemd 模板和环境变量模板。
- `plugins/`
  - AI 协作插件入口。

## 核心主链

### 1. 入口与装配

- `internal/entry/`
  - 把 config、session、model、runtime、server 装配成可运行服务。
- `internal/config/`
  - 负责配置结构、YAML 读取、环境变量覆盖和校验。

### 2. Transport 层

- `internal/server/`
  - HTTP、SSE、OpenAPI、control-plane transport 和请求映射。
  - `chat/stream`、`chat/respond`、`runtime/respond` 都从这里进入。

### 3. 应用编排层

- `internal/app/`
  - 负责请求编排、等待态、补数、deferred queue、control-plane use case 和模型治理 use case。

### 4. 运行时层

- `internal/runtime/`
  - 负责任务归一化后的运行时执行主链。
- `internal/sandbox/`
  - 定义 core runtime 与外部 sandbox 执行边界之间的最小稳定契约，当前用于 `external_sandbox_ref` validation result。
- `internal/runtime/task/`
  - 统一任务输入模型。
- `internal/runtime/scene/`
  - 场景匹配和 scene catalog。
- `internal/runtime/intent/`
  - 意图解析和 route 决策。
- `internal/runtime/persona/`
  - persona 相关运行时消费。
- `internal/runtime/store/`
  - runtime store 边界。

### 5. 持久化与治理

- `internal/session/`
  - session store，支持 memory/postgres。
- `internal/model/`
  - provider/model 治理和模型接入。
- `internal/model/parameters/`
  - 模型参数策略中心。
- `internal/controlplane/`
  - scene、skill、tool、governance 和 config version 管理。

### 6. 能力层

- `internal/skills/`
  - builtin skill 与 uploaded package 治理。
- `internal/tools/`
  - 原子 tool 与 registry。
- `internal/validationmcp/`
  - Validation layer 的轻量 `athena-validation-mcp` 适配器，用于验证 tool schema、governance decision、安全调用结果和 redacted trace。
- `internal/workflow/`
  - workflow plan 生成。
- `internal/inspection/`
  - inspection 相关结构和逻辑骨架。
- `internal/alerts/`
  - alerts 相关结构和逻辑骨架。
- `internal/automation/`
  - automation 相关结构和逻辑骨架。
- `internal/knowledge/`
  - 知识检索与候选相关结构和逻辑骨架。
- `internal/memory/`
  - 记忆相关结构与辅助。

### 7. 专项增强与基础设施

- `internal/extensions/`
  - 让平台专项增强和主线分离。
- `internal/extensions/platform/`
  - 目前承载 platform 上下文、自动化 payload、tool descriptor 等专项增强。
- `internal/observability/`
  - 统一动作日志和观测字段。
- `internal/policy/`
  - 治理和策略相关基础类型。
- `internal/postgresutil/`
  - PostgreSQL 辅助封装。
- `internal/runtimeassets/`
  - runtime builtin 资源和 registry。

## `web/` 模块

- `web/src/api.ts`

## `scripts/` 模块

当前分三类：

- 联调和可靠性验证：
  - `chat_stream_reliability.py`
  - `chat_respond_reliability.py`
  - `agent_platform_smoke.py`
- 交付与部署：
  - `build_release_bundle.sh`
  - `deploy_cloud.sh`

## `deploy/` 模块

- `docker-compose.cloud.yml`
  - 单仓云端 Docker Compose 部署样例。
- `athena.env.example`
  - 云端部署环境变量模板。
- `systemd/athena.service.example`
  - 二进制部署用的 systemd 单元模板。

## 一条典型请求怎么走

以 `POST /api/chat/stream` 为例：

1. `main.go`
2. `internal/entry`
3. `internal/server/http.go`
4. `internal/app`
5. `internal/runtime`
6. 需要时进入：
   - `internal/runtime/intent`
   - `internal/runtime/scene`
   - `internal/extensions/platform`
   - `internal/tools`
   - `internal/skills`
7. 回到 `internal/server` 输出 SSE：
   - `progress_step`
   - `stream_chunk`
   - `done`

## 修改代码时的默认落点

- 改 HTTP / SSE 协议：
  - `internal/server/`
- 改请求编排、waiting、queue：
  - `internal/app/`
- 改任务归一化、scene、intent、执行主链：
  - `internal/runtime/`
- 改 platform 特有增强：
  - `internal/extensions/platform/`
- 改治理和后台配置：
  - `internal/controlplane/`
- 改模型 provider / model：
  - `internal/model/`
- 改 session store：
  - `internal/session/`
